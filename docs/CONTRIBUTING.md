# Contributing to Kryton Plugins

Thank you for your interest in contributing plugins to Kryton! This guide covers how to add a new plugin to the registry.

## Prerequisites

- Node.js 24+
- npm
- Git
- A working knowledge of TypeScript

## Getting Started

1. **Clone the repository:**
   ```bash
   git clone https://github.com/azrtydxb/kryton-plugins.git
   cd kryton-plugins
   npm install
   ```

2. **Copy the plugin template:**
   ```bash
   cp -r templates/plugin-template plugins/my-plugin
   ```

3. **Edit the manifest:**
   Update `plugins/my-plugin/manifest.json` with your plugin's details. The `id` field must match the directory name.

4. **Implement your plugin:**
   - Server plugins: edit `server/index.ts` (TypeScript, built to CJS)
   - Client plugins: edit `client/index.ts` (TypeScript, built to ESM)
   - Type definitions are in `types/server.d.ts` and `types/client.d.ts`
   - You can have both, or just one

5. **Build your plugin:**
   ```bash
   npm run build
   ```
   This compiles `.ts` files to `.js` using esbuild. The manifest still references `.js` files.

## Directory Structure

```
plugins/my-plugin/
  manifest.json           # Required: plugin metadata
  server/
    index.ts              # Optional: server entry point (TypeScript, compiled to CJS)
  client/
    index.ts              # Optional: client entry point (TypeScript, compiled to ESM)
```

## Manifest Requirements

All manifests must include these fields:

| Field | Description |
|-------|-------------|
| `id` | Must match the directory name exactly |
| `name` | Human-readable display name |
| `version` | Semver version (must match the version in `registry.json`) |
| `description` | Brief description |
| `author` | Your name |
| `minKrytonVersion` | Minimum Kryton version required |

At least one of `server` or `client` entry points must be declared.

## Testing Locally

Run all checks before submitting:

```bash
# Build TypeScript plugins to JS
npm run build

# Type-check all plugins
npm run typecheck

# Lint all plugin JS files
npm run lint

# Validate registry and manifests
npm run validate

# Test plugin exports and syntax (requires build first)
npm run test
```

Fix any linting issues automatically:

```bash
npm run lint:fix
```

## Code Style

- **All plugins**: TypeScript with proper type annotations
- **Server plugins**: Import types from `types/server.d.ts`
- **Client plugins**: Import types from `types/client.d.ts`
- Use `const` and `let`, never `var`
- Use strict equality (`===` / `!==`)
- Always use semicolons
- Follow the ESLint rules configured in `eslint.config.js`

## PR Workflow

1. **Create a branch:**
   ```bash
   git checkout -b add-my-plugin
   ```

2. **Make your changes:**
   - Add your plugin directory under `plugins/`
   - Add your plugin entry to `registry.json`
   - Ensure version in manifest matches version in registry

3. **Run all checks:**
   ```bash
   npm run build && npm run typecheck && npm run lint && npm run validate && npm run test
   ```

4. **Commit and push:**
   ```bash
   git add plugins/my-plugin
   git commit -m "feat: add my-plugin"
   git push -u origin add-my-plugin
   ```
   Note: Built `.js` files and `registry.json` are auto-generated by CI.

5. **Open a Pull Request:**
   - CI will automatically run lint, validate, and test checks
   - All checks must pass before merge
   - Describe what your plugin does in the PR description

## Review Criteria

Pull requests are evaluated on:

- **Correctness**: Plugin passes all validation and tests
- **Quality**: Clean code, follows style guidelines
- **Documentation**: Manifest has accurate description and settings
- **Security**: No access to data outside the plugin's scope
- **Compatibility**: Correct `minKrytonVersion` specified
